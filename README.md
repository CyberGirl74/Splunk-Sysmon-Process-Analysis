# Splunk Sysmon Process Analysis

## Overview

This project demonstrates how I used Splunk Enterprise and Microsoft Sysmon to analyze Windows process creation events using Sysmon Event ID 1. The objective was to gain hands-on experience with Splunk Search Processing Language (SPL), investigate parent-child process relationships, identify common Windows processes, and strengthen foundational blue team investigation skills.

---

## Objectives

- Configure Splunk to ingest Sysmon logs
- Analyze Windows Event ID 1 (Process Creation)
- Practice writing SPL queries
- Investigate parent-child process relationships
- Identify frequently executed processes
- Develop hands-on blue team investigation skills

---

## Tools Used

- Splunk Enterprise 10.4
- Microsoft Sysmon

# Investigation 1 – Parent Process Analysis for PowerShell

### Purpose

This query identifies which parent processes launched PowerShell. Parent-child process relationships are an important part of threat hunting because attackers frequently abuse PowerShell to execute malicious commands.

### SPL Query

```spl
index=* EventCode=1
Image="*powershell.exe"
| stats count by ParentImage
| sort -count
```

### Findings

The results showed that most PowerShell executions originated from **splunkd.exe**, indicating that Splunk itself launched PowerShell during indexing and monitoring activities.

Other observed parent processes included:

- WindowsTerminal.exe
- explorer.exe

These are expected parent processes and represent normal Windows user activity rather than suspicious behavior.

# Investigation 2 – Top Process Creation Events

### Purpose

This query identifies the most frequently executed processes captured by Sysmon Event ID 1. Reviewing common processes helps establish a baseline of normal system behavior and assists analysts in identifying unusual activity.

### SPL Query

```spl
index=* EventCode=1
| top Image limit=10
```

### Findings

The results revealed that the majority of recorded events were generated by legitimate Splunk background services. This is expected because Splunk continuously monitors, indexes, and processes log data.

Frequently observed processes included:

- splunk-optimize.exe
- splunk-powershell.exe
- python3.9.exe
- cmd.exe
- splunkd.exe
- postgres.exe
- splunk-regmon.exe
- splunk-netmon.exe
- splunk-admon.exe
- splunk-MonitorNoHandle.exe

These are legitimate Splunk Enterprise components responsible for indexing, monitoring, database services, and system management.

## Lessons Learned

This project strengthened my understanding of Windows process creation events and demonstrated how Splunk can be used to analyze Sysmon logs using SPL. I gained experience identifying parent-child process relationships, establishing a baseline of normal system activity, and recognizing the difference between expected system processes and potentially suspicious behavior.

Although this investigation did not reveal malicious activity, it reinforced an important cybersecurity principle: security analysts must first understand what "normal" looks like before they can accurately identify anomalies.
